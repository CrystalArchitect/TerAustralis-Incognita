# ADR-0004: Lock the CrystalCore naming taxonomy; ban future CrystalCore-* runtime names

**Status:** Accepted
**Date:** 2026-07-23

## Context

"CrystalCore" already names five different things in this repository: the
Lumina framework, the protocol pack, CrystalBridge (whose Python package is
literally also named `crystalcore`), the mythos terminal
(`CrystalCore.OS`), and the repository/governance platform itself
(`CrystalCore OS` — see [`docs/vision/CrystalCore.md`](../vision/CrystalCore.md),
written to disambiguate exactly this).

In review of the v1.0 reorganization (PR #25, PR #27), a v0.2 proposal
suggested a new runtime component under the working name "CrystalCore
Engine." Adding a sixth referent for the same word would make the
disambiguation problem worse at the exact moment the project is trying to
define new architecture — the review itself flagged this as the question
to answer before any module design: *what is CrystalCore*, before *how is
it implemented*.

## Decision

- **Adopt a canonical four-branch taxonomy** for "CrystalCore": CrystalCore
  Framework (Lumina's engine), CrystalCore Protocol (the protocol pack),
  CrystalBridge (the integration layer), and CrystalCore OS (this
  repository's platform architecture). Every existing use of the name maps
  onto one of these branches —
  [`docs/vision/CrystalCore.md`](../vision/CrystalCore.md) is the map, kept
  current as the source of truth.
- **The mythos terminal's name (`CrystalCore.OS`) is left as-is, not
  folded into the taxonomy or silently renamed.** It collides with the
  platform's name almost exactly, which is precisely the confusion this ADR
  exists to prevent going forward — but resolving a pre-existing collision
  is a separate, deliberate decision, not a side effect of this one. It's
  documented as the taxonomy's one open case rather than papered over.
- **No future runtime component becomes a fifth or sixth "CrystalCore."**
  A new component gets a name that describes its role. Candidates raised in
  review: *Crystal Runtime*, *Crystal Nexus*, *Crystal Coordinator*,
  *Crystal Kernel* — this ADR doesn't pick one; it only rules out growing
  the same overloaded word further.
- This rule already governed the AI Orchestrator concept
  ([`ADR-0005`](ADR-0005.md)), which was named for its role from the start
  rather than becoming "CrystalCore AI Router."

## Consequences

- Anyone proposing a new component now has a naming rule to check against
  before writing a spec, not after.
- The pre-existing mythos-terminal/platform collision remains — readers hit
  it via [`docs/vision/CrystalCore.md`](../vision/CrystalCore.md)'s explicit
  callout rather than being surprised by it.
- If a future decision renames the terminal or the platform to resolve that
  collision, it needs its own ADR; this one doesn't pre-authorize a choice.
