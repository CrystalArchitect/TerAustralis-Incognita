# ADR-0011: Adopt the three-project boundary model — umbrella, Crystal Core, Crystal Vision

**Status:** Accepted
**Date:** 2026-07-23

Per [`Decision-Records.md`](../governance/Decision-Records.md), this ADR is
accepted when the maintainer merges the PR carrying it; until then it is a
proposal.

## Context

Two things forced this decision.

**First, the docs and the repositories disagree.** This repository's
entry-point documentation described a `src/` code tree, `scripts/`, and
`tests/` that have never existed in its git history on any branch — the
code lives in the maintainer's local working tree, with a dated copy under
`archive/2026/local-snapshot-2026-07-17/`. The drift is measurable: merged
commit `1c1e473`'s message describes rewriting seven
`packages/*/LICENSE.md` files, but its diffstat touches none — those edits
existed only on the local disk. CI (`ci.yml`) and the Pages deploy
(`deploy.yml`) fail against the absent paths. The companion commit to this
ADR (the "sync entry-point docs with repository reality" commit) records
that reality in the docs; see
[`SystemMap.md`](../architecture/SystemMap.md#where-the-code-actually-lives).

**Second, the project needed its boundaries said out loud before anything
moves.** A second repository, `TerAustralis-Incognita-Code`, already exists
(one commit, a one-line README) with no charter, and the maintainer has
directed a document-first sequence: synchronize docs with reality, define
boundaries, produce a migration plan — and only after that approval,
discuss renames, moves, or new repositories.

The maintainer's directives, recorded verbatim in intent:

1. **TerAustralis Incognita** is the umbrella: governance, architecture
   docs, ADRs, and references to other repos — not the main app code.
2. **Crystal Core** owns the engine, runtime, APIs, and shared libraries.
3. **Crystal Vision** is the user-facing application built on top.
4. **Lumina**, the flagship companion, sits **wholly in Crystal Vision**
   for now — no framework split unless a strong reason emerges later.
5. **Clementine** — the orchestration layer coordinating comms between AI
   systems — is a **logical component**, not a separate repo.

Constraints that bound this decision: the Constitution's §1 locked names
(TerAustralis Incognita · CrystalVision · CrystalCore.Lattice) stay locked;
the CrystalCore naming taxonomy is fixed by
[`ADR-0004`](ADR-0004.md); the Incognita Rule requires every claim below to
be labeled honestly; and Repository-Principle 4 requires the whole thing to
stay incremental and reversible.

## Decision

1. **Three projects, one umbrella.** The project family is bounded as:
   - **TerAustralis Incognita (umbrella)** — canon, governance,
     architecture documentation, ADRs, mythos, research, archive, and
     references to the engineering repositories. It holds no main app code.
   - **Crystal Core** — the engineering project owning the engine,
     runtime, APIs, and shared libraries.
   - **Crystal Vision** — the engineering project owning the user-facing
     application built on Crystal Core.
2. **Lumina belongs wholly to Crystal Vision**, including its embedded
   CrystalCore Framework package, which keeps its ADR-0004 name while
   traveling with the app. Criteria for a future framework split are
   operationalized in the Migration Plan, not left as vibes.
3. **Clementine is a named logical component inside Crystal Core** — the
   orchestration layer coordinating communications between AI systems
   (today: the Starline Weaver hub, `clementine/bridge/` in the protocol
   pack). It is not a repository and does not become one by drift.
4. **The standing charter is
   [`Project-Boundaries.md`](../governance/Project-Boundaries.md)** — the
   component→project map, the dependency rule (Crystal Vision may depend
   on Crystal Core; never the reverse), and what stays in the umbrella.
   **[`Migration-Plan.md`](../governance/Migration-Plan.md) is a labeled
   proposal** — staged, each stage separately approvable and reversible,
   nothing executed by this ADR.
5. **`TerAustralis-Incognita-Code` is reserved by charter** as the
   engineering repository; it stays empty by design until a Migration-Plan
   stage is approved. The repo-count question (one engineering repo vs.
   separate core and vision repos) is explicitly deferred to that plan's
   decision point.
6. **Decided-NOT list**, so nobody has to infer it: no directory renames,
   no file moves, no new repositories, no workflow changes, no
   Constitution edits, no locked-name redefinition, and no change to the
   ADR-0004 taxonomy. "Crystal Core the project" *contains* the CrystalCore
   Framework, CrystalCore Protocol, and CrystalBridge branches; the
   CrystalCore OS platform remains the umbrella's repository-architecture
   concept; the CrystalCore OS / CrystalCore.OS collision stays exactly as
   ADR-0004 left it — resolving it would need its own ADR.
   This ADR also supersedes
   [`REPO-RESTRUCTURING-PLAN.md`](../governance/REPO-RESTRUCTURING-PLAN.md)
   as the operative restructuring proposal (that document's `packages/`
   split was reverted by [`ADR-0010`](ADR-0010.md) and its layout never
   landed in this repository's history).

## Consequences

- The entry-point docs now tell the truth about what is in this repository,
  and the boundary language ("umbrella", "engine", "user-facing") finally
  has one written referent instead of several implied ones.
- `ci.yml` and `deploy.yml` remain red — deliberately. Their fate is a
  Migration-Plan stage (workflow changes were excluded from this change
  set), so the failure is now documented rather than mysterious.
- The Full-Stack-v0.5 product map's branding of `crystal-interface` under
  Crystal Core is superseded by the charter's placement of both demo
  shells under Crystal Vision as *applications* — the operator/citizen
  distinction survives in their descriptions; brand-facing is not code
  ownership.
- Anything that later contradicts the charter needs an ADR; locked names
  can only move via Constitution §8. The migration itself remains
  unauthorized until the maintainer approves stages one at a time.
